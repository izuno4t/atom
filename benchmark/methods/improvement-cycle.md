# 評価から再評価までの改善サイクル

この文書は、外部ツール比較を使って `atom` の変換品質を改善する
基本手順を定義する。

`benchmark/` は MarkItDown など外部ツールとの比較と事前棚卸しに使う。
`evaluation/` は `atom` 本体のfixture評価と回帰検出に使う。

## 目的

評価の目的は、単に他ツールより速いかどうかを確認することではない。
次の順序で、改善すべき原因を特定し、実装へ戻し、再評価する。

1. 入力集合を固定する。
2. 比較対象ツールでMarkdown化し、成功と失敗を棚卸しする。
3. 同じ入力集合を `atom` でMarkdown化する。
4. 成功可否、処理時間、文字数、Markdownハッシュ、警告を比較する。
5. 差分を原因分類へ落とす。
6. 原因に対する最小の改善を実装する。
7. 同じ入力集合で再評価する。
8. 判断をGit管理対象の文書へ記録する。

## 保存場所

| 種別 | 保存先 | Git管理 |
| ---- | ---- | ---- |
| 対象ディレクトリ設定 | `benchmark/benchmark.config.toml` | 管理外 |
| 外部ツール出力 | `benchmark/outputs/<tool>/` | 管理外 |
| `atom` 出力 | `benchmark/outputs/atom/` | 管理外 |
| TSVレポート | `benchmark/reports/` | 管理外 |
| 判断サマリ | `docs/` または `benchmark/methods/` | 管理対象 |

対象ディレクトリや実文書の絶対パスは、Git管理対象ファイルへ保存しない。
再現に必要な対象数、条件、分類結果だけを管理対象の文書へ残す。

## 手順

### 1. 入力集合を固定する

入力集合は `benchmark/benchmark.config.toml` で指定する。
検証では、指定されたディレクトリ直下の通常ファイルだけを扱い、
サブディレクトリは辿らない。

最初に全体を探索してから個別処理するのではなく、ファイル単位で処理し、
既存TSVに記録済みのパスはスキップできるようにする。これにより、
途中停止やtimeout後も早いファイルから順に結果を蓄積できる。

### 2. 比較対象ツールで棚卸しする

MarkItDownなどの比較対象ツールで、対象をすべてMarkdown化する。
各ファイルについて最低限、次を記録する。

- `path`
- `status`
- `elapsed_ms`
- `chars`
- `sha256`
- `error`

`status` は、少なくとも `ok`、`empty`、`error`、`timeout` を区別する。
`ok` だけを成功扱いにし、`empty` は「処理は終わったがMarkdown本文がない」
ものとして分ける。

### 3. `atom` で同じ棚卸しをする

同じ入力集合を同じ粒度で `atom` に通す。
比較の前に、実行したバイナリを特定できるよう `atom --version` を確認する。

```bash
target/release/atom --version
```

`atom` 側も比較対象ツールと同じ列を持つTSVを出す。
出力Markdownはハッシュだけでなく、必要に応じて人が直接読めるよう
`benchmark/outputs/atom/` に保存する。

### 4. 比較対象を分ける

改善対象は、少なくとも次の集合へ分ける。

- 比較対象ツールが成功し、`atom` が失敗したもの
- 比較対象ツールが成功し、`atom` も成功したもの
- 比較対象ツールが失敗し、`atom` が成功できる可能性があるもの
- 両方が失敗し、入力側の制約が疑われるもの

同じ対策で直る集合と、原因が異なる集合を混ぜない。
例えば、PDFの文字復元失敗、OCR必須、権限制限、Office XML構造の未対応は
別の原因として扱う。

### 5. 原因分析する

原因分析では、Markdown出力だけを見て判断しない。
入力ファイル、警告、処理ログ、report、比較対象ツールの挙動を合わせて見る。

PDFでは次を分ける。

- 抽出可能なテキスト層があるが、文字復元に失敗している
- CIDや埋め込みフォントのUnicode対応が足りない
- ページ画像またはアウトライン化文字のみで、OCRが必要
- 暗号化またはStandard security handlerによる権限制限がある
- 表、段組、スライド由来レイアウトの読順復元が弱い

Office文書では次を分ける。

- OOXML packageの部品解決が足りない
- relationship解決が足りない
- sharedStrings、notes、commentsなど文書種別固有部品が足りない
- 表、結合セル、図表、captionのMarkdown表現が弱い
- writer側のMarkdown整形で情報が落ちている

## 改善方法の基本方針

### 外部ツールは正解ではなく参照実装として扱う

MarkItDown、Pandoc、Docling、PyMuPDF4LLMなどの出力は比較材料であり、
正解ではない。正解は入力文書を人が確認し、Markdownとして保持すべき
構造を決めて判断する。

外部CLIを通常変換の必須依存にはしない。通常の `atom` 変換は、
インストール済みバイナリ単体で動くことを前提にする。

### バックエンド選定を改善方針にしない

失敗のたびにバックエンドを切り替える方針にはしない。
PDF抽出器やその後段の整形は `atom` の中核機能なので、外部ツールの
Markdown出力へ委譲せず、`atom` 側で原因ごとに改善する。

Rust crateは、PDF構造、ストリーム展開、演算子走査などの低レベル部品として
使う。文字復元、座標付き単語化、行復元、表復元、読順復元は `atom` 側の
責務として扱う。

### 失敗理由を出力品質の一部として扱う

変換できない入力を、単に `error` で終わらせない。
少なくとも次のどれかへ分類し、人が次の判断をできる状態にする。

- OCRが必要
- 権限制限がある
- Unicode mapが不足している
- 現在の抽出器では未対応
- 入力形式として対象外

読めない理由を正しく列挙できることは、Markdown本文を出力できることとは
別の品質である。成功率だけでなく、失敗理由の説明可能性も評価する。

### 改善は最小単位で入れる

実装は原因単位で分ける。
例えば、CID補完、画像PDF判定、security handler判定、読順復元、writer整形を
同じ変更に混ぜない。

変更ごとに、次を満たす。

- 失敗を再現する最小テストを追加する
- 原因に対応する実装だけを変える
- 同じ入力集合で再評価する
- 改善対象外の集合に回帰がないか確認する

### 閾値や評価関数で失敗を隠さない

評価関数の閾値を下げる、timeoutを広げる、対象を外す、expectedを根拠なく
更新する、といった方法で失敗を隠さない。

対象外にする場合は、入力側の制約、セキュリティ上の理由、ライセンス、
OCR未設定など、理由を文書に残す。

## 再評価と記録

改善後は、同じ対象集合で再評価する。
対象集合を変えた場合は、前後比較として扱わない。

管理対象の文書には次を残す。

- 実施日
- 対象条件
- 対象数
- 成功数
- 失敗理由の内訳
- 処理時間の概況
- 代表的な低品質出力の確認結果
- 採用した改善方針
- 次に改善すべき原因

`benchmark/reports/` はGit管理外なので、最終判断は必ず `docs/` か
`benchmark/methods/` のMarkdownへ要約する。

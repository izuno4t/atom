# TASKS

マイルストーン: M6
ゴール: 実コーパスをディレクトリ単位で評価し、原因分類と改善優先度を判断できる状態にする

## ワークフロールール

- タスク着手時にステータスを 🚧 に更新する
- タスク完了時にステータスを ✅ に更新する
- DependsOn のタスクがすべて ✅ でないタスクには着手しない

## ステータス表記ルール

| Status | 意味 |
| ---- | ----- |
| ⏳ | 未着手、TODO |
| 🚧 | 作業中、IN_PROGRESS |
| 🧪 | 確認待ち、REVIEW |
| ✅ | 完了、DONE |
| 🚫 | 中止、CANCELLED |

## 管理方針

- M1-M5の旧タスク表は `docs/archive/tasks-m1-m5.md` にアーカイブする。
- M1-M5から残った未完了タスクは `docs/tasks-m7.md` に移動する。
- 現行 `docs/tasks.md` はM6の評価プロセスだけを管理する。
- 外部ツール比較、事前棚卸し、ディレクトリ別評価は `benchmark/` に置く。
- `evaluation/` は `atom` 本体のfixture評価と回帰検出に限定する。
- 入力ルートや実文書の絶対パスはGit管理対象へ保存しない。
- ディレクトリ別タスクにはディレクトリ名だけを保存する。

### TASK-028

- 補足: `TASK-024` で最優先と判定した共通原因を修正する。
- 対象: 複数ディレクトリで発生し、成功率または出力品質への影響が最も大きい
  原因。
- 成果: 原因に対応する最小修正、回帰テスト、対象サンプルでの確認結果を残す。
- 注意: ほかの原因を同じ変更に混ぜない。
- 結果: MarkItDownとの比較対象を、`atom` が扱える拡張子だけに絞るよう
  棚卸し処理を修正した。TASK-028時点の比較対象270件の再評価では、
  MarkItDownが `ok` で `atom` が `error` の負けケースは0件になった。
- 結果: `.md`、`.txt`、`.csv`、`.xml`、`.svg`、`.gdoc`、`.xlsm`、
  PDF互換`.ai` を既存経路に接続し、PDF座標異常によるpanicを捕捉するよう
  修正した。
- 結果: TASK-028時点の `ok` / `ok` 263件では、MarkItDown合計698,797msに対し
  `atom` は425,048msだった。速度負け107件の遅延合計59,142msのうち、PDFが
  82件、57,395msを占める。
- 判断: `.zip`、`.key`、`.reg` は通常文書変換の比較対象ではないため
  supported拡張子から外した。直接入力時は unsupported document を返してよい。
- 判断: `31_67.pdf` の速度外れ値に対し、`pdf_extract` を小型PDFで先行させる案を
  実測したが、29.9秒かかりpanic文も出たため採用しなかった。

### TASK-029

- 補足: `TASK-028` の修正結果を前提に、次点の共通原因を修正する。
- 対象: 最優先原因の次に件数、影響範囲、修正可能性が高い原因。
- 成果: 原因に対応する最小修正、回帰テスト、対象サンプルでの確認結果を残す。
- 注意: `TASK-028` の回帰がないことを確認してから着手する。
- 結果: `TASK-028` 後の次点原因はPDF速度外れ値と判断した。TASK-029着手時点の
  MarkItDown勝ち107件の遅延合計59,142msのうち、PDFが82件、57,395msを占めた。
- 結果: 最大外れ値 `31_67.pdf` に対して `pdf_extract` を小型PDFで先行させる
  小修正案を実測したが、29.9秒かかりpanic文も出たため採用しなかった。
- 結果: supported PDF集合で候補順序を測定し、`raw-content` 常時先行は
  `RM-Final.pdf` を悪化させるため採用しなかった。2MiB以下PDFに限定して
  `raw-content` を `pdf-rs` より先に試す条件では、実在PDF 192件で usable が
  191件から192件、timeoutが1件から0件、合計時間が294,681msから271,820msへ
  改善した。
- 結果: `31_67.pdf` はMarkItDown 1,766ms、`atom` 2,968msとなり、修正前の
  `atom` 31,968msから改善した。
- 判断: PDF速度外れ値はM6内で改善済みとし、単一PDFではなくsupported PDF集合の
  成功状態、速度分布、timeout、外れ値を根拠に採用判断した。
